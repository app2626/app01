## Architecture

Only 4 source files in `src/` — all frontend code is inlined into the served HTML via `<?!= include('CSS'); ?>` / `<?!= include('JS'); ?>` in `Index.html`:

1. **`Code.js`** (backend, ~1720 lines) — `doGet` reads the app name from Settings key `SystemName` (falls back to "Mobile Pre Order System") and serves `Index.html` with `XFrameOptionsMode.DEFAULT` (anti-clickjacking; only switch back to ALLOWALL for intentional iframe embeds). **Every client-server call goes through `apiHandler(action, payload, userToken)`**, a switch-based router; the frontend wraps it as `window.app.api(action, payload)` over `google.script.run` **with a 45s `Promise.race` timeout** (google.script.run has none of its own — never remove this). Auth: HMAC-SHA256 tokens (`generateToken`/`verifyToken` with `timingSafeEqual` signature compare), 30-day expiry, signed with a key auto-generated into Script Properties (`MPOS_SECRET_KEY`) — never hardcode a fallback secret. Login is rate-limited (5 fails / 5 min per username via CacheService `LOGIN_FAIL_<username>`). `apiHandler` re-loads the user from the `Members` sheet on every call (`secureUser`); never trust role/branch from the client.
2. **`Index.html`** (~1200 lines) — app shell + every screen as a `<template id="tmpl-...">` (`login`, `main`, `dashboard`, `pos`, `analytics`, `target`, `bookingsummary`, `salessummary`, `allbookingsummary`, `datagrid`, `invoice`). Loads Tailwind (CDN, `darkMode: 'class'`), Font Awesome, Chart.js + datalabels plugin, SweetAlert2, html2canvas. Modals carry `role="dialog" aria-modal="true"`. `<base target="_top">` is required so external links/forms escape the GAS sandbox iframe — see rule 16. A no-flash dark-mode boot script sits at the very top of `<head>`, right after `<meta charset>` — see the 2026-07-21 entry above and rule 26.
3. **`JS.html`** (frontend, ~5720 lines) — everything hangs off global `window.app`. Sub-modules: `pos` (cart/checkout), `dashboard`, `analytics`, `target`, `bookingSummary`, `salesSummary`, `allBookingSummary`, `grid` (generic CRUD data grid driven by column configs in `route()`), `invoice`, `systemSettings`, `settingsManager`. Hash-based routing in `window.app.route()` — also re-invoked by `toggleTheme()` to force charts/tables to re-render in the new theme. Session: `localStorage` key `tg_pos_user` (auto-migrates from a stale `sessionStorage` copy; both cleared on logout); cart: `localStorage` key `tg_cart`; remembered booking staff: `sessionStorage` `tg_bk_staff`/`tg_bk_phone`; theme choice: `localStorage` key `tg_theme`.
4. **`CSS.html`** (~726 lines) — custom CSS over Tailwind (flat cards, KPI gradients, cart-drawer 3-zone flex layout, print styles, `:focus-visible` ring, dark-mode variable + utility-class overrides). Theme tokens live in `:root`/`html.dark` CSS variables (`--color-*`, `--radius-*`, `--shadow-*`) — use them instead of new hardcoded colors, and keep the flat/Sarabun aesthetic (see Project Overview). See rule 26 before adding new hardcoded Tailwind color classes.

### Frontend helpers (use these — don't reinvent)

- `window.app.esc(v)` — HTML-escape; mandatory for any sheet/user data in `innerHTML`/attributes.
- `window.app.escAttrJs(v)` — for values embedded as JS string literals inside `onclick="fn('...')"`.
- `window.app.highlightEsc(raw, term)` — escape + wrap search matches in `<mark>`; safe for display text only, never for `title=`/`alt=` attributes.
- `window.app.emptyState({icon, title, subtitle, actionLabel, onclick})` — the standard empty/error state for every page (grid empty/search-miss/load-error, POS no-results, dashboard/analytics/target setup prompts). Error states pair "โหลดข้อมูลไม่สำเร็จ" with a "ลองใหม่" button that re-calls the page's `init`. `onclick` must be trusted literal code — never user data.
- `window.app.buttonBusy(btn, text)` — disable + spinner for submit buttons awaiting API; returns a restore fn that **must** be called in `finally`; returns `null` if already busy (re-entry guard). `window.app.iconBtnBusy(btn)` — same idea for small icon-only buttons (no text, would overflow).
- `window.app.selectImageForUpload(inputEl, stateKey, onLocalPreview)` / `window.app.flushImageUpload(stateKey)` — deferred image upload: selecting a file validates it and stores it in `pendingImageUploads[stateKey]` with a local object-URL preview only; the real `UPLOAD_IMAGE` call happens when the Save handler calls `flushImageUpload`. Known stateKeys: `'gridImage'`, `'banner-<rowIndex>'`, `'newBanner'`, `'loginBg'`, `'invoiceLogo'`, `'approverSign'`. Do not reintroduce an immediate-upload-on-select pattern.
- `window.app.chartExportButtons(canvasId)` — call once right after `new Chart(...)`; injects a download/copy button pair via `Chart.getChart(canvasId)`. `window.app.tableExportButtons(cardEl)` — same idea for non-canvas summary tables (goes through `html2canvas` on an off-screen clone with `sticky`/`overflow` stripped so the export captures the full table, not just the visible scroll window). Both are idempotent — safe to call on every re-render.
- `window.app.toggleTheme()` — flips dark/light, persists to `localStorage`, then calls `window.app.route()` to force the current page (and its charts) to re-render under the new theme. `window.app.applyTheme(theme)` — the non-toggling half (sets the `dark` class, `meta[theme-color]`, `Chart.defaults`, syncs toggle-button icons); called directly at boot in `init()` and after `renderLogin()`/`renderLayout()` rebuild the DOM. `window.app.chartTheme()` — returns `{text, grid, datalabel}` for the *current* theme; pass its values into any new `new Chart(...)` config instead of hardcoding hex colors (see rule 26).

### Data model (Google Sheets)

Sheets: Products, Members, Branches, Channels, Promotions, Interests, GiftMappings, AutoPromotions, Orders, OrderStatus, InventoryLog, AuditLog, Settings, UI_Banners, Target, Invoices. Always address columns **by header name, never by index**.

- **Orders**: one logical order = **multiple rows sharing an `OrderID`** (main items + gift rows + `DISCOUNT` rows). Customer fields (and `Client Request ID`) are filled **only on the first row** of an order. Schema (24 columns, `ORDERS_HEADERS_` constant in `Code.js`, also used by `setupDatabase`): `OrderID, Timestamp, Channel, Branch Code, Customer Name, Contact Number, Email, ID Card_Passport, Code Handraiser, SKU, Product Name, Qty, Unit Price, Promo, Reservation Status, Staff, Booking Phone, Customer Interests, Remark, Row Total, Order Status, Receipt No, Deposit, Client Request ID` — **always append new columns at the END**; `processCheckout` auto-adds missing trailing headers to older sheets, then **hard-blocks checkout** if the sheet's header order doesn't match `ORDERS_HEADERS_` position-for-position (see rule 25 — `orderRows.push()` writes positional arrays, not header-mapped, so a manually reordered column would otherwise silently misfile data).
- **Idempotency**: the client sends `clientRequestId` with CHECKOUT. Server scans the `Client Request ID` column inside the lock; on a hit it returns `{status:'success', orderId:<existing>, duplicate:true}` without writing a duplicate row.
- **AutoPromotions**: `Rule ID` (e.g. `AP-001`) is the primary key — do not key off `Buy Category`, it isn't unique. Has a `Channel` condition column (multi-select comma list, added 2026-07-20) — blank/`*`/`ALL` = applies to every channel, otherwise tokenize-compared against the order's actual channel, same pattern as `GiftMappings.Channel` (rule 20). Checked in `processCheckout`'s Auto Bundle ceiling calc, `pos.checkAutoPromotions`, and the POS bundle badge in `pos.cardHtml`.
- **OrderStatus**: `Status ID / Status Name / Color Code` — drives the read-only status timeline in the Orders edit modal. Row order = flow order (Cancelled excluded; rendered separately). Fallback flow if sheet has <2 rows: Pending → Confirmed → Paid → Delivered.
- **Settings**: key-value rows. `ReserveStart`/`ReserveEnd` drive the POS booking window (enforced **server-side** in `processCheckout` too) plus Analytics/Target pages. Written as plain text via `setNumberFormat('@')` so Sheets can't coerce them into Dates. `saveSettingsItem`'s `type:'bookingsettings'` path is a **generic key-value writer**, not booking-specific — `SystemName`, `DriveFolderId`, `Invoice*` (company info), `ApproverName`, `ApproverSignatureUrl` all go through it.
- **Target**: column 1 matches an Area / Branch Code / Branch Name / Staff name; `ALL` or `ALL BRANCHES` = grand total of mobile sales. ⚠️ If a sheet has both an `ALL`/`ALL BRANCHES` row and individual branch/area rows, the `target` module's grand-total footer double-counts (sums every row including the already-aggregated one) — known issue, not yet fixed, see "Known parked issues".
- **Invoices** (`INVOICES_HEADERS_` in `Code.js`): one document = one row, line items as JSON in `Items JSON`. Status flow `รออนุมัติ` → `อนุมัติแล้ว` (Admin/Manager approve; approved docs are locked — Admin can still delete). `Approved By` is stamped from Settings key `ApproverName`, not the clicking user (the clicker is in `AuditLog`).

### Roles

**Admin**: everything. **Manager**: most pages; GiftMappings/AutoPromotions as readonly grids; on Orders can only change `Order Status`, cannot delete; can approve invoices. **Sales**: locked to own Branch/Channel (POS selects pre-filled + disabled), read-only Orders grid, no settings. `Accessible Menus` on a member overrides role defaults — matched **exactly per comma-separated menu id**, never substring matching. Enforce every permission in `Code.js` too, not just in the UI: `GET_TABLE` allows `Members`/`Settings`/`AuditLog` to Admin only (and always strips `Password`); `GET_SETTINGS_LIST` is Admin-only; `SAVE_RECORD` rejects `Orders` entirely (use `UPDATE_FULL_ORDER`); Sales only sees own-branch rows of Orders/InventoryLog; Invoices are filtered to `Requested By === secureUser.Username` for non-Admin/Manager.

## Critical Rules (learned from real bugs — verify still true before relying on them for anything security-sensitive)

1. **Timestamps are display strings.** `getTableDataAsJson` uses `getDisplayValues()`, so `Timestamp` is the sheet's display format, never ISO. Parse with `new Date(str)` then build a **local** `YYYY-MM-DD` key. **Never use `toISOString()` for date keys, date-input defaults, or export filenames** — Thailand is UTC+7 and dates shift back a day. Numbers from sheets may contain commas — strip with `.replace(/,/g,'')` before `parseInt`/`parseFloat` (Stock, Price, Row Total all bit this codebase before).
2. **Escape everything injected into HTML.** Use `window.app.esc(value)` for any sheet/user data placed into `innerHTML` or attribute strings; `escAttrJs()` inside onclick literals. Exception: AutoPromotions `Message Suggest`/`Message Apply` are intentionally rendered as HTML.
3. **Never trust the client in `processCheckout`.** Price/name/category come from the `Products` sheet only; Qty must be an integer ≥ 1; category `โมบาย` capped at 1 unit/order; จอง T requires `receiptNo` + `depositAmount > 0`; booking window checked server-side; every discount row is re-validated against `Promotions` (Percent promos cap at pct% of the sheet-priced goods total) and Auto Bundle discounts are recomputed from `AutoPromotions`.
4. **Gift out-of-stock must not block the sale.** `processGift` logs `"GIFT (NO STOCK)"` qty 0 to `InventoryLog` and marks the gift order row "รอสต๊อกของแถม" — the ledger reflects reality, the mobile sale still completes.
5. **PII is written as plain text.** `Contact Number` and `ID Card_Passport` are written unobfuscated by `processCheckout`/`updateFullOrder`, with `setNumberFormat('@')` on those columns first (or Sheets coerces phone strings to numbers and drops leading zeros). **Read paths keep `deobfuscate()`** (`GET_TABLE` Orders, dashboard) for backward compat with old base64-encoded rows (digit-passthrough returns plain values untouched) — never remove it. One-time migration for legacy rows: `migrateDeobfuscatePII()`, run manually from the GAS editor (not exposed via `apiHandler`). Never return the `Password` field to the client from any endpoint.
6. **Data grid indices.** Rows render from `filteredDataList` (search/sort/pagination applied). Any per-row action must use the absolute index into `filteredDataList`, never the page-local index or `dataList`. Export CSV also exports `filteredDataList`.
7. **Caching (CacheService, 6h TTL).** `Products, Promotions, Branches, Channels, Interests, GiftMappings, AutoPromotions` are cached as `TABLE_<name>`. `saveRecord`/`deleteRecord` clear the edited table's cache, and **every stock write** (`processCheckout`, `updateFullOrder`, `deleteRecord` Orders path) must `CacheService.getScriptCache().remove("TABLE_Products")` — otherwise POS oversells against stale stock. Direct sheet edits won't show for up to 6h unless cleared; adding a column to a cached sheet requires clearing its cache.
8. **Atomic writes.** Every write op (`processCheckout`, `updateFullOrder`, `saveRecord`, `deleteRecord`, settings saves, invoice CRUD) holds `LockService.getScriptLock()` with `waitLock(10000)` and releases in `finally`. Batch writes: build arrays in memory, write once with `Range.setValues()` — never `appendRow`/`setValue` in a loop. Stock updates: read `Products` once, mutate in memory, write the `Stock` column back in one call.
9. **Checkout payload field names are fixed:** `cart, channel, branch, customerName, contactPhone, email, idCard, codeHandraiser, customerInterests, resStatus, bkStaffName, bkPhone, remark, promo, discount, discounts[], receiptNo, depositAmount, clientRequestId`. Do not invent aliases (`items`, `customer`, `phone`, `total` do not exist). `printReceipt` computes totals from `pay.cart` itself.
10. **Checkout submit flow:** `isSubmitting`/button disable should be released **immediately after CHECKOUT returns**, before any post-success work; a `finally` should only be a safety net. Post-sale refresh work should live in its own try/catch that only `console.warn`s — the order is already saved, so a refresh failure must never surface an error dialog or block the next order.
11. **POS validation gates:** cart non-empty → channel selected → branch selected (checked separately). จอง T requires `receiptNo` + `depositAmount > 0`; จอง F does not. Category `โมบาย` is limited to 1 unit per order (enforced in add-to-cart AND qty updates AND server-side). Out-of-stock cards render without their onclick (plus a gray badge) — keep both layers.
12. **Error contract:** backend returns `{status:'success'|'error', message}`; frontend must always check and surface failures — via `Swal.fire` for actions, or the standard `emptyState` error card (with retry) for page loads. No silent catches.
13. **Changing `getSecretKey` / `MPOS_SECRET_KEY` invalidates all sessions** — users must re-login once.
14. **Never put `/*` inside any string/template literal in `JS.html` or inline `<script>` code.** GAS strips JS comments from `<script>` blocks **at serve time**, and its stripper treats `/*` inside a string as a comment opener — it can swallow everything up to EOF, drop the closing `</script>`, and white-screen the app. Use explicit MIME lists (`accept="image/png,image/jpeg,image/webp,image/gif"`) instead of `image/*`. `node --check` passes on such code — it only breaks after deploy, so grep for `/\*` in `JS.html` strings before pushing.
15. **UI conventions:** decorative absolutely-positioned layers (blur orbs, overlays, badges) must have `pointer-events-none`. Images: `.skeleton` container + `opacity-0` img + `onload` fade-in. Icon-only buttons need a Thai `aria-label`. Thai strings in `Code.js` must stay UTF-8 — if you see `���` mojibake, rewrite the string.
16. **Never let internal `#hash` nav links fall through to default anchor behavior.** `Index.html` has `<base target="_top">` (needed so genuine external links/forms escape the GAS sandbox iframe), but that also means a bare `<a href="#pos" class="nav-link">` click would navigate the **top-level** frame instead of just changing the hash — this tears down and re-embeds the sandboxed app iframe and can leave a permanent white screen after any menu click post-login. The fix is a delegated listener in `init()` (`JS.html`) that does `e.preventDefault()` on `a.nav-link[href^="#"]` clicks and sets `window.location.hash` manually — do not remove it, and route any new internal nav link through `.nav-link` (or its own `preventDefault`).
17. **Changing `oauthScopes` in `appsscript.json` does not reliably trigger a new consent prompt.** Re-running any function from the GAS editor after a scope change may show no "Authorization required" dialog, even though the deployed `/dev` web app still throws on the newly-required scope. Don't assume "I added the scope and re-authorized" means the deployed app actually has it — verify by testing the real feature on `/dev`. If it's stuck, fully revoke the project's access at myaccount.google.com/permissions and re-authorize from scratch.
18. **Tailwind CDN's cascade order is later than it looks.** `CSS.html` is included near the top of `Index.html`, but the Tailwind CDN `<script>` sits near the bottom and injects its generated `<style>` into `<head>` when it runs — so Tailwind's utility classes land **later** in the cascade than custom `CSS.html` rules despite appearing earlier in the document. Any custom rule targeting a property a Tailwind class also sets on the same element needs `!important` or it silently loses. Also: adding a **new named color** to `window.tailwind.config.theme.extend.colors` may not render at all in this app (classes like `bg-bronze-500` can produce no CSS even with a full shade set), while overriding an *existing* scale (e.g. `indigo`) works fine — root cause unknown; use arbitrary-value syntax (`bg-[#c19a4b]`) as the reliable workaround. **Never name a custom CSS class after a Tailwind utility keyword** (`inline`, `flex`, `hidden`, `block`, `relative`, `absolute`, etc.) — it can silently collide with Tailwind's own utility of the same name (Tailwind loads after `CSS.html` and wins).
19. **The cart drawer's 3-zone flex layout (`CSS.html` `#cartDrawer`/`.cart-drawer-header`/`.cart-drawer-content`/`.cart-drawer-footer`) has interdependent fixed sizes.** Header is `flex: 0 0 60px`, footer is `flex: 0 0 (50px + safe-area)`, and content's explicit height calc must stay in sync with the footer's actual height — the drawer's whole point is that content can never scroll under the header or footer. If you change the footer's height, update content's calc in the same edit or the "no overlap ever" guarantee breaks silently.
20. **Any field that can hold a multi-select comma list (currently: GiftMappings `Channel`, saved by the admin grid's `multi_select_channel` UI as e.g. `"Retail, Online"`) must be compared by splitting into trimmed-lowercase tokens and checking membership — never `=== wholeString` (exact match always fails once >1 option is selected) and never `wholeString.includes(candidate)` (substring match can false-positive across unrelated values, e.g. `"RetailPlus, Online".includes("Retail")`). `processCheckout`'s `channelMatches` (`Code.js`) and `pos.updateModalWithProduct`'s `chMatch` (`JS.html`) both do this correctly as of 2026-07-19 — if you add another multi-select-backed match anywhere (new field, new page), follow the same tokenize-then-`.includes(token)` pattern, and keep client and server logic in sync since the client's picker and the server's checkout validation must agree or a user can select something the server later rejects.
21. **Cart stock checks must sum across all cart lines of the same SKU, not just the line being touched.** The same SKU can appear as multiple separate cart lines when different gift combinations are picked (the add-to-cart dedup key includes `brandGifts`/`channelGifts`). `pos.confirmAddToCart`/`pos.updateQtyInCart` compute a cross-line total (`cartData.filter(i=>i.SKU===sku)...reduce`) before allowing a qty increase — checking only the single line's qty against `stockOf(p)` lets the cart overcommit stock that only surfaces as a late, confusing error at server checkout.
22. **Never embed real `<input>`/`<file>` form fields inside `Swal.fire({html: ...})` for a primary data-entry flow.** Confirmed unreliable 2026-07-20 fixing the Hero Banners/Promo Grid/Popup "เพิ่มแบนเนอร์ใหม่" add-new flow (`settingsManager` in `JS.html`): the old `openAddModal()` built its form as an HTML string handed to `Swal.fire({html})`, and live testing showed the Image URL field's placeholder rendering as garbled literal markup plus unreliable saves (`SAVE_SETTINGS_ITEM` could silently fail to persist). Use an inline toggle-card instead — a `hidden` `<div>` with real ids rendered directly into the page (not through SweetAlert), shown/hidden via `classList.remove/add('hidden')`, saved by a plain function reading `document.getElementById(...)` directly. This is exactly the pattern the per-row `แก้ไข` edit-cards already use (proven reliable) and what `settingsManager.addItem`/`cancelAddItem` now use for the add-new form. `Swal.fire` is still fine for confirmations, alerts, and validation messages (`Swal.showValidationMessage`, plain `Swal.fire('...', '...', 'warning')`) — the failure mode is specifically real form `<input>` elements living inside `html:`.
23. **Verify you're pushing from `pbzf8.2/`, not the repo root, before `clasp push -f`.** `../.clasp.json` (repo root `app01/`) and `pbzf8.2/.clasp.json` point at the **identical `scriptId`**, but their `src/` trees are independent copies that can silently diverge — discovered 2026-07-20 when `app01/src` was found missing all 9 fixes from the 2026-07-19 session (some earlier push presumably ran from the wrong directory). Whichever directory is pushed last wins and silently overwrites the other's deployed code with no warning from `clasp`. Confirm `pwd` (or `cat .clasp.json`) resolves inside `pbzf8.2/` before pushing; never `cd` to `app01/` root and push for this scriptId. This divergence has not been cleaned up (the two trees still differ) — don't assume `app01/src` is safe to read from or push.
24. **Any user-typed free-text value written to a Sheet (`Customer Name`, `Remark`, `Staff`, invoice customer/company fields, etc.) must go through `sanitizeSheetText_()` (`Code.js`) before `setValues()`.** A cell string starting with `=`/`+`/`-`/`@` is interpreted by Sheets as a formula **regardless of the column's number format** — the codebase's existing `setNumberFormat('@')` calls only stop numeric/date auto-coercion (e.g. phone leading-zero loss), they do **not** stop formula injection from an API write. `sanitizeSheetText_` prepends a literal `'` (Google/OWASP's standard mitigation) when the value starts with one of those characters. Applied at every free-text write site in `processCheckout`, `updateFullOrder` (only to the specific known free-text header names — never blanket-apply to all headers, or numeric fields like `Row Total` get stringified and break sums), and `saveInvoice`. Also related: the discount-ceiling checks in `processCheckout` (`maxAutoTotal`/`promoUsed`) accumulate usage **across the whole `discounts[]` array** per Auto-Bundle-vs-named-promo, not per individual row — checking each row against the full ceiling independently (the pre-2026-07-20 behavior) let a crafted payload with duplicate discount rows multiply the allowed discount, since a normal checkout only ever sends one Auto-Bundle row plus distinct-named promo rows (see `pos.confirmCheckout`'s payload assembly in `JS.html`).
25. **`processCheckout`'s `orderRows.push()` calls write fixed-position 24-element arrays, not header-name-mapped** (unlike `updateFullOrder`, which correctly maps by header name) — so `Code.js` hard-blocks checkout with a clear error if the live `Orders` sheet's header row doesn't match the `ORDERS_HEADERS_` constant position-for-position, rather than silently writing a column's data into the wrong field (e.g. PII landing in `Customer Interests`). This is intentionally a **hard block, not a soft warning** — confirmed with the owner 2026-07-20 that the live sheet's column order is trusted to match. If the `Orders` sheet is ever intentionally restructured outside `setupDatabase`, update `ORDERS_HEADERS_` in the same change or checkout will start failing immediately. Never insert/reorder a column in the `Orders` sheet directly from the Sheets UI.
26. **Dark mode (added 2026-07-21) is class-based (`html.dark` on `<html>`), not automatic, and has three independent theming layers that don't sync themselves.** (1) Anything already styled via `var(--color-*)` (chart-card, kpi-gradient-*, swal2, login-card, buttons, cart-drawer, the new `.sidebar-toggle-btn`/`.sidebar-show-handle`) re-themes for free from the single `:root`/`html.dark` variable block at the top of `CSS.html`'s dark-mode section. (2) Everything else is a hardcoded Tailwind utility class used directly in `Index.html`/`JS.html` markup (`bg-white`, `text-slate-600`, `border-slate-200`, etc.) — these do **not** read CSS variables and need their own `html.dark .<class> { ... !important }` rule in `CSS.html`'s dark-mode `@media screen` block (same cascade-fighting pattern as rule 18). **A new hardcoded color utility class you introduce will not theme automatically just because it's Tailwind** — check that block for the class first, and add a rule if it's missing. (3) Chart.js is a separate case again: charts paint to `<canvas>`, not CSS, so colors come from `window.app.chartTheme()` passed in at chart-creation time (there are only 4 `new Chart(...)` call sites total in the app — grep for it before assuming you need a 5th pattern), and `window.app.toggleTheme()` forces a full `window.app.route()` re-render rather than live-repainting an existing Chart.js instance (existing charts already destroy-and-recreate themselves on re-render, so this is safe). The whole utility-class override block is scoped inside `@media screen` — never let dark-mode rules leak into `@media print`, since invoices/reports must always print on white paper regardless of the logged-in user's theme choice.

27. **Capacity strings in this app are `"RAM/StorageUnit"` format (e.g. `"12/256GB"`, `"16/1TB"`), not a bare size** — any regex/parser that extracts a capacity's numeric size for sorting or grouping must require the `TB|GB|MB` unit suffix to immediately follow the matched number, or it will grab the leading RAM figure instead (e.g. matching `"12"` out of `"12/256GB"` and ranking every capacity variant of a model as tied). `bookingSummary.render`'s `capVal` (`JS.html`, added 2026-07-24 for capacity row sorting) uses `/([\d.]+)\s*(TB|GB|MB)\b/i` with a **required** unit group for this reason — this is a different regex from the pre-existing `capVal` in the `pos` module (~line 3900, used for the POS capacity filter dropdown), which has an *optional* unit group and works fine there only because those capacity values don't have a RAM prefix. Don't copy one `capVal` implementation over the other assuming they're interchangeable — check whether the input can contain a leading non-capacity number first.

28. **Whether a gift row actually had stock deducted at checkout must be read from that row's own `Remark` marker (`"รอสต๊อกของแถม"`, set by `processGift` when the gift was out of stock — rule 4), never inferred from current product stock levels.** Any code that reverses a gift row's stock effect later (order delete, order cancel, or any future "undo"-shaped path) has to check `Remark === 'รอสต๊อกของแถม'` on that specific row before crediting stock back — a heuristic like "is current stock less than this row's qty" can be wrong in both directions, since stock may have moved (restocked, sold elsewhere) between checkout and the later delete/cancel action. `deleteRecord`'s Orders branch and `updateFullOrder`'s `adjustStockAndLog` REVERT half both do this correctly as of 2026-07-25 — if you add another stock-reversal path, follow the same pattern (check the marker, don't guess from stock numbers).

## Known parked issues

- **Target & Forecast grand-total double-counts if the `Target` sheet mixes an `ALL`/`ALL BRANCHES` row with individual branch/area rows** (`JS.html` `target` module, ~line 2495-2524). The footer sums every row's `actualVal` unconditionally, but the `ALL` row's value is already the grand total of every other row combined — with both present, the footer overstates by roughly 2x. **Not fixed** — the correct fix depends on what the owner intends `ALL BRANCHES` to mean (a pure aggregate row that should be excluded from the footer sum, vs. a genuinely separate company-wide target that doesn't equal the sum of the listed branches). Ask before touching; don't guess.
- **GiftMappings edit-form gift checkboxes can wrongly pre-tick a similarly-named product when reopening a saved mapping for edit** (`JS.html` `grid` module, ~line 4761-4773, `multi_select_brand_gifts`/`multi_select_channel_gifts` read-back). The pre-check uses substring matching against previously-saved exact product names (e.g. a mapping saved with `"เคสกันกระแทก"` would also pre-tick `"เคสกันกระแทก สีดำ"` if that product now exists). **Not fixed** — tightening to exact-match is the obvious correct-looking fix but could silently break any legacy row that was hand-typed as a genuine keyword rather than picked via checkboxes; needs to confirm no such rows exist first.
- **`p.Channel.includes(currentChannel)` substring match in the POS product filter + variant picker** (`JS.html` ~line 3408/3832) has the same false-positive-prone shape as the two comma-list bugs fixed 2026-07-19 (rule 20), but this one is a **known, previously owner-declined fix** in the sibling project `preorder3.2` ("fix not yet approved, leave as-is" — affects `Channel='ALL'` product semantics). Don't fix here without first confirming the owner wants this specific one changed now — it's not an oversight, it was a conscious call.

## Common Workflows

**Add a column to Orders:** append to END of the schema in `setupDatabase` → extend the header auto-add block in `processCheckout` and append the value to **all three** `orderRows.push()` calls (main item, gift, discount — arrays are positional and must stay the same length; currently 24 elements ending in `Client Request ID`) → add a column config to the Orders `grid.init` call in `JS.html` **only if it should be user-visible** → add to `printReceipt` if it prints.

**Add a new page:** `<template id="tmpl-xxx">` in `Index.html` → menu entry in `window.app.menuConfig` (top of `JS.html`) with allowed roles → module `window.app.xxx = { init, ... }` + a case in `window.app.route()` → if it needs backend data, add an action to `apiHandler` (enforce role there too) → use `emptyState` for empty/error/setup states and `buttonBusy` on its submit buttons.

**Porting a fix to/from `../preorder3.2`:** the two projects share lineage but have no shared branch — verify the target line actually differs before assuming a fix is missing or already present (see the deltas listed in Project Overview).
