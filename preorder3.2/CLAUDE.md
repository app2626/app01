## Architecture

Only 4 source files in `src/` — all frontend code is inlined into the served HTML via `<?!= include('CSS'); ?>` / `<?!= include('JS'); ?>` in `Index.html`:

1. **`Code.js`** (backend, ~1600 lines) — `doGet` serves `Index.html` with `XFrameOptionsMode.DEFAULT` (anti-clickjacking; only switch back to ALLOWALL for intentional iframe embeds). **Every client-server call goes through `apiHandler(action, payload, userToken)`**, a switch-based router; the frontend wraps it as `window.app.api(action, payload)` over `google.script.run` **with a 45s `Promise.race` timeout** (google.script.run has none of its own — never remove this). Auth: HMAC-SHA256 tokens (`generateToken`/`verifyToken` with `timingSafeEqual` signature compare) signed with a key auto-generated into Script Properties (`MPOS_SECRET_KEY`) — never hardcode a fallback secret. Login is rate-limited (5 fails / 5 min per username via CacheService `LOGIN_FAIL_<username>`). `apiHandler` re-loads the user from the Members sheet on every call (`secureUser`); never trust role/branch from the client.
2. **`Index.html`** — app shell + every screen as a `<template id="tmpl-...">`. Loads Tailwind (CDN), Font Awesome, Chart.js + datalabels plugin, SweetAlert2, html2canvas (2026-07-12, for the table image-export feature — see Project Overview). Modals carry `role="dialog" aria-modal="true"`; a global Esc handler (end of JS.html) closes the topmost layer in order popupBanner → formModal → checkoutModal → addToCartModal → cart drawer.
3. **`JS.html`** (frontend, ~5600 lines) — everything hangs off global `window.app`. Sub-modules: `pos` (cart/checkout), `dashboard`, `analytics`, `target`, `grid` (generic CRUD data grid driven by column configs in `route()`), `invoice`, `systemSettings`, `settingsManager`. Hash-based routing in `window.app.route()`. **Settings are one tabbed page** (round 9): `#systemsettings` (Admin only) renders a POS-style underline tab bar with 6 tabs — booking window, hero banners, promo grid, popup, login BG, auto setup; each tab body is `settingsManager.init(type, 'ssTabContent')` (the pre-round-9 pages, re-parented via `settingsManager.containerId`, default `'page-content'`). The 6 legacy hashes (`#bookingsettings` `#herobanners` `#promogrids` `#popupbanners` `#loginbg` `#autosetup`) are gone from `menuConfig` but `route()` redirects them (via `location.replace`, **before** the access check — they'd fail the menu lookup) to `#systemsettings` opening the matching tab via `systemSettings.pendingTab`; the tab id list in `systemSettings.tabs` is the single source for the redirect map and the permission aliases. Session: `sessionStorage` key `tg_pos_user` (deliberately not localStorage); cart: localStorage key `tg_cart`; remembered booking staff: sessionStorage `tg_bk_staff`/`tg_bk_phone` — all three cleared on logout.
4. **`CSS.html`** — custom CSS over Tailwind (glassmorphism, skeleton shimmer, KPI gradients, cart-drawer 3-zone flex layout, print styles, `:focus-visible` ring). Theme tokens live in `:root` CSS variables (`--color-*`, `--shadow-*`) — use them instead of new hardcoded colors.

### Frontend helpers (use these — don't reinvent)

- `window.app.esc(v)` — HTML-escape; mandatory for any sheet/user data in `innerHTML`/attributes.
- `window.app.escAttrJs(v)` — for values embedded as JS string literals inside `onclick="fn('...')"`.
- `window.app.highlightEsc(raw, term)` — escape + wrap search matches in `<mark>`; safe for display text only, never for `title=`/`alt=` attributes.
- `window.app.emptyState({icon, title, subtitle, actionLabel, onclick})` — the standard empty/error state for every page (grid empty/search-miss/load-error, POS no-results, dashboard `#dashEmptyState`, analytics/target setup prompts). Error states pair "โหลดข้อมูลไม่สำเร็จ" with a "ลองใหม่" button that re-calls the page's `init`. `onclick` must be trusted literal code — never user data.
- `window.app.buttonBusy(btn, text)` — disable + spinner for submit buttons awaiting API; returns a restore fn that **must** be called in `finally`; returns `null` if already busy (re-entry guard). Settings save fns locate their button via `document.activeElement`.
- `window.app.selectImageForUpload(inputEl, stateKey, onLocalPreview)` / `window.app.flushImageUpload(stateKey)` — deferred image upload (see Project Overview): selecting a file validates it and stores it in `pendingImageUploads[stateKey]` with a local object-URL preview only; the real `UPLOAD_IMAGE` call happens when the Save handler calls `flushImageUpload`. Known stateKeys: `'gridImage'`, `'banner-<rowIndex>'`, `'swalBanner'`, `'loginBg'`.
- `window.app.chartExportButtons(canvasId)` — call once right after `new Chart(...)`; injects a download/copy button pair into the canvas's `.chart-card` header via `Chart.getChart(canvasId)`, so it works no matter which module's registry holds the instance. `window.app.tableExportButtons(cardEl)` — same idea for non-canvas summary tables; call once per `.chart-card` after setting its container's `innerHTML`. Both are idempotent (check for an existing `.chart-export-btns`/`.table-export-btns` node before inserting) so it's safe to call on every re-render. Underlying pieces: `downloadChartImage`/`copyChartImage` (canvas → `toBase64Image`/`toBlob`), `downloadTableImage`/`copyTableImage`/`renderTableToCanvas` (DOM → `html2canvas` on an off-screen clone with `sticky`/`overflow` stripped — see Project Overview), `chartTitleOf(el)` (finds the nearest ancestor `<h6>` for the download filename), `iconBtnBusy(btn)` (like `buttonBusy` but icon-only, no text — don't reuse `buttonBusy` on a 26px button, the default "กำลังบันทึก..." text will overflow it).

### Data model (Google Sheets)

Sheets: Products, Members, Branches, Channels, Promotions, Interests, GiftMappings, AutoPromotions, Orders, OrderStatus, InventoryLog, AuditLog, Settings, UI_Banners, Target, Invoices. Always address columns **by header name, never by index**.

- **Orders**: one logical order = **multiple rows sharing an `OrderID`** (main items + gift rows + `DISCOUNT` rows). Customer fields (and `Client Request ID`) are filled **only on the first row** of an order. Schema ends with `..., Order Status, Receipt No, Deposit, Client Request ID` — **always append new columns at the END**; `processCheckout` auto-adds missing trailing headers to older sheets.
- **Idempotency**: the client sends `clientRequestId` (`pos.pendingRequestId`) with CHECKOUT. It is created once per order attempt, kept across retries/timeouts, cleared on server-confirmed success **and reset in `saveAndRender()`** (cart change = new attempt). Server scans the `Client Request ID` column inside the lock; on a hit it returns `{status:'success', orderId:<existing>, duplicate:true}` without writing — the client shows a toast instead of the success dialog.
- **OrderStatus**: `Status ID / Status Name / Color Code` — drives the read-only status timeline in the Orders edit modal (`grid.orderTimelineHtml`). Row order = flow order (Cancelled excluded; it renders as a separate red banner). If the sheet has <2 rows the fallback flow is Pending → Confirmed → Paid → Delivered, and an unknown current status is appended as the final step.
- **Settings**: key-value rows. `ReserveStart`/`ReserveEnd` drive the POS booking window (enforced **server-side** in `processCheckout` too), Analytics, and Target pages (those pages show the standard emptyState setup prompt if unset). Booking settings are written as plain text via `setNumberFormat('@')` so Sheets can't coerce them into Dates. `saveSettingsItem`'s `type:'bookingsettings'` path is a **generic key-value writer**, not booking-specific despite the name — `InvoiceCompanyName`/`InvoiceCompanyAddress`/`InvoiceCompanyEmail`/`InvoiceLogoUrl` (invoice page defaults, set under ตั้งค่าระบบ → ฐานข้อมูล) and `DriveFolderId` both go through the same path.
- **Target**: column 1 matches an Area / Branch Code / Branch Name / Staff name; `ALL` or `ALL BRANCHES` = grand total of mobile sales.

### Roles

**Admin**: everything. **Manager**: most pages; GiftMappings/AutoPromotions as readonly grids; on Orders can only change `Order Status`, cannot delete. **Sales**: locked to own Branch/Channel (POS selects pre-filled + disabled), read-only Orders grid, no settings. `Accessible Menus` on a member overrides role defaults — matched **exactly per comma-separated menu id** (never substring matching); the 6 legacy settings ids (e.g. `bookingsettings`) count as aliases granting `systemsettings` in `checkMenuAccess` — keep that, old member rows rely on it. Enforce every permission in `Code.js` too, not just in the UI: `GET_TABLE` allows `Members`/`Settings`/`AuditLog` to Admin only (and always strips `Password`); `SAVE_RECORD` rejects `Orders` entirely (use `UPDATE_FULL_ORDER`); Sales only sees own-branch rows of Orders/InventoryLog.

## Critical Rules (learned from real bugs)

1. **Timestamps are display strings.** `getTableDataAsJson` uses `getDisplayValues()`, so `Timestamp` is the sheet's display format, never ISO. Parse with `new Date(str)` then build a **local** `YYYY-MM-DD` key. **Never use `toISOString()` for date keys, date-input defaults, or export filenames** — Thailand is UTC+7 and dates shift back a day. Numbers from sheets may contain commas — strip with `.replace(/,/g,'')` before `parseInt`/`parseFloat` (Stock, Price, Row Total all bit us).
2. **Escape everything injected into HTML.** Use `window.app.esc(value)` for any sheet/user data placed into `innerHTML` or attribute strings (grid cells, form `value=""`, datalists, receipts, banners, `<option>`s); `escAttrJs()` inside onclick literals. Exception: AutoPromotions `Message Suggest`/`Message Apply` are intentionally rendered as HTML.
3. **Never trust the client in `processCheckout`.** Price/name/category come from the Products sheet only; Qty must be an integer ≥ 1; category `โมบาย` capped at 1 unit/order; จอง T requires `receiptNo` + `depositAmount > 0`; booking window checked server-side; every discount row is re-validated against Promotions (Percent promos cap at pct% of the sheet-priced goods total) and Auto Bundle discounts are recomputed from AutoPromotions.
4. **Gift out-of-stock must not block the sale.** `processGift` logs `"GIFT (NO STOCK)"` qty 0 to InventoryLog and marks the gift order row "รอสต๊อกของแถม" — the ledger reflects reality, the mobile sale still completes.
5. **PII is written as plain text (owner's decision, 2026-07-05).** `Contact Number` and `ID Card_Passport` are written unobfuscated by `processCheckout`/`updateFullOrder` so the owner can read the sheet directly — with `setNumberFormat('@')` on those columns first, or Sheets coerces phone strings to numbers and drops leading zeros. **Read paths keep `deobfuscate()`** (`GET_TABLE` Orders, dashboard) for backward compat with old base64 rows (digit-passthrough returns plain values untouched) — never remove it. One-time migration for old rows: run `migrateDeobfuscatePII()` manually from the GAS editor (Admin; not exposed via `apiHandler`). Never return the `Password` field to the client from any endpoint.
6. **Data grid indices.** Rows render from `filteredDataList` (search/sort/pagination applied). Any per-row action must use the absolute index into `filteredDataList` (`startOffset + index` from `renderTable`), never the page-local index or `dataList`. Export CSV also exports `filteredDataList`.
7. **Caching (CacheService, 6h TTL).** `Products, Promotions, Branches, Channels, Interests, GiftMappings, AutoPromotions` are cached as `TABLE_<name>`. `saveRecord`/`deleteRecord` clear the edited table's cache, and **every stock write** (`processCheckout`, `updateFullOrder`, `deleteRecord` Orders path) must `CacheService.getScriptCache().remove("TABLE_Products")` — otherwise POS oversells against stale stock. Direct sheet edits won't show for up to 6h unless cleared; adding a column to a cached sheet requires clearing its cache.
8. **Atomic writes.** Every write op (`processCheckout`, `updateFullOrder`, `saveRecord`, `deleteRecord`, settings saves) holds `LockService.getScriptLock()` with `waitLock(10000)` and releases in `finally`. Batch writes: build arrays in memory, write once with `Range.setValues()` — never `appendRow`/`setValue` in a loop. Stock updates: read Products once, mutate in memory, write the Stock column back in one call.
9. **Checkout payload field names are fixed:** `cart, channel, branch, customerName, contactPhone, email, idCard, codeHandraiser, customerInterests, resStatus, bkStaffName, bkPhone, remark, promo, discount, discounts[], receiptNo, depositAmount, clientRequestId`. Do not invent aliases (`items`, `customer`, `phone`, `total` do not exist). `printReceipt` computes totals from `pay.cart` itself.
10. **Checkout submit flow (hotfix — don't regress):** `isSubmitting`/button disable is released **immediately after CHECKOUT returns**, before any post-success work; the `finally` is only a safety net. The post-sale refresh (`GET_ALL_DATA` + `pos.filter()`) lives in its own try/catch that only `console.warn`s — the order is already saved, so it must never surface an error dialog or block the next order.
11. **POS validation gates:** cart non-empty → channel selected → branch selected (all three, separately). จอง T requires `receiptNo` + `depositAmount > 0`; จอง F does not. Category `โมบาย` is limited to 1 unit per order (enforced in add-to-cart AND qty updates AND server-side). Out-of-stock cards render without their onclick (plus a gray badge) — keep both layers.
12. **Error contract:** backend returns `{status:'success'|'error', message}`; frontend always checks and surfaces failures — via `Swal.fire` for actions, or the standard `emptyState` error card (with retry) for page loads. No silent catches.
13. **Changing `getSecretKey` / MPOS_SECRET_KEY invalidates all sessions** — users must re-login once.
14. **Never put `/*` inside any string/template literal in `JS.html` or inline `<script>` code.** GAS strips JS comments from `<script>` blocks **at serve time**, and its stripper treats `/*` inside a string as a comment opener — it swallowed everything from `accept="image/*"` to EOF, the closing `</script>` disappeared, the browser dropped the entire unterminated block, and the app white-screened (2026-07-07, took a full evening to diagnose). Use explicit MIME lists (`accept="image/png,image/jpeg,image/webp,image/gif"`). `node --check` passes on such code — it only breaks after deploy, so grep for `/\*` in JS.html strings before pushing.
15. **UI conventions:** decorative absolutely-positioned layers (blur orbs, overlays, badges) must have `pointer-events-none` (a blur overlay once made the login form untappable). Images: `.skeleton` container + `opacity-0` img + `onload` fade-in. Banner preload sizes in `preloadBanners()` must match render sizes (`w1600` hero, `w1000` grid) or the cache warm-up is useless. Icon-only buttons need a Thai `aria-label`. Thai strings in `Code.js` must stay UTF-8 — if you see `���` mojibake, rewrite the string.
16. **Never let internal `#hash` nav links fall through to their default anchor behavior.** `Index.html` has `<base target="_top">` (needed so any genuine external link/form escapes the GAS sandbox iframe), but that also means a bare `<a href="#pos" class="nav-link">` click navigates the **top-level** frame instead of just changing the hash in place — this makes Google's wrapper tear down and re-embed the sandboxed app iframe (visible as a `userCodeAppPanel?createOAuthDialog=true` request), which sometimes never finishes and leaves a permanent white screen after any menu click post-login (root-caused and fixed 2026-07-08). The fix is a single delegated listener in `init()` (`JS.html`, right after the `hashchange` listener) that does `e.preventDefault()` on `a.nav-link[href^="#"]` clicks and sets `window.location.hash` manually — do not remove it, and do not add new internal nav links without going through `.nav-link` + this handler (or their own `preventDefault`).
17. **Changing `oauthScopes` in `appsscript.json` does not reliably trigger a new consent prompt.** Re-running any function from the GAS editor after a scope change may show no "Authorization required" dialog at all, even though the deployed `/dev` web app still throws an authorization error on the newly-required scope. If this happens the user must fully revoke the project's access at myaccount.google.com/permissions and re-authorize from scratch. Don't assume "I added the scope and re-authorized" means the deployed app actually has it; verify by testing the real feature on `/dev`. (Beware the reverse mistake too: the 2026-07-08 Drive `setSharing` failure *looked* like a scope problem for days but was actually permission inheritance — see the resolved entry in Known parked issues.)
18. **Tailwind CDN's cascade order is later than it looks.** `CSS.html` is included near the top of `Index.html` (`<?!= include('CSS'); ?>`), but the Tailwind CDN `<script src="https://cdn.tailwindcss.com">` sits near the bottom and injects its generated `<style>` into `<head>` when it runs — so Tailwind's utility classes land **later** in the cascade than custom CSS.html rules despite appearing earlier in the document. Any custom CSS.html rule targeting a property a Tailwind class also sets on the same element (e.g. `padding-bottom` via a custom class vs Tailwind's `p-2`; `bottom` via a custom `#id` selector vs Tailwind's `bottom-6`) needs `!important` or it silently loses. This bit the 2026-07-09 responsive pass (`.cart-drawer-footer` safe-area padding, `#posCartFab` safe-area bottom offset both needed `!important`). Related Tailwind CDN quirk (2026-07-10 theme retone): **adding a NEW named color** (e.g. `bronze:`) to `window.tailwind.config.theme.extend.colors` did not render at all in this app — classes like `bg-bronze-500` produced no CSS (blank white boxes) even with a full 50–900 shade set, while overriding the EXISTING `indigo` scale works fine. Root cause unknown; workaround is arbitrary-value syntax (`bg-[#c19a4b]`, `from-[#c19a4b]`), which works reliably. The premium theme (2026-07-10, deployed @32): `indigo` scale remapped to slate-navy (#366891/#254d71/#1b3a59...), CSS variables in CSS.html `:root` updated to match, bronze/gold accents on the 4 brand logo marks + login-page blob via arbitrary hex values, `#dataTable th` header gradient and Swal confirm hover now navy, PWA `theme-color` #254d71. **Also never name a custom CSS class after a Tailwind utility keyword** (`inline`, `flex`, `hidden`, `block`, `relative`, `absolute`, etc.) — the 2026-07-12 chart/table export feature named a modifier class `.inline` and it silently collided with Tailwind's own `.inline{display:inline}` utility (same cascade-order issue as above: Tailwind loads after CSS.html and won the property), turning a flex button row into a broken vertical stack. Fixed by renaming to `.table-export-btns`.
19. **The cart drawer's 3-zone flex layout (`CSS.html` `#cartDrawer`/`.cart-drawer-header`/`.cart-drawer-content`/`.cart-drawer-footer`) has interdependent fixed sizes.** Header is `flex: 0 0 60px`, footer is `flex: 0 0 (50px + safe-area)`, and content's explicit `height: calc(100dvh - 60px - 50px - safe-area)` must stay in sync with whatever the footer's actual height is — the drawer's whole point (per its own comment block) is that content can never scroll under the header or footer. If you change the footer's height (e.g. adding `env(safe-area-inset-bottom)`), update content's calc in the same edit or the drawer's "no overlap ever" guarantee breaks silently.

## Common Workflows

**Add a column to Orders:** append to END of the schema in `setupDatabase` → extend the header auto-add block in `processCheckout` and append the value to **all three** `orderRows.push()` calls (main item, gift, discount — arrays are positional and must stay the same length; currently 24 elements ending in `Client Request ID`) → add a column config to the Orders `grid.init` call in `JS.html` **only if it should be user-visible** (internal columns like `Client Request ID` are deliberately omitted) → add to `printReceipt` if it prints.

**Add a new page:** `<template id="tmpl-xxx">` in `Index.html` → menu entry in `window.app.menuConfig` (JS.html top) with allowed roles → module `window.app.xxx = { init, ... }` + a case in `window.app.route()` → if it needs backend data, add an action to `apiHandler` (enforce role there too) → use `emptyState` for empty/error/setup states and `buttonBusy` on its submit buttons.
